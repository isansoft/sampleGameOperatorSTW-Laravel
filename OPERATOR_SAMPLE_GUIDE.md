# Prime Mac Games Operator Sample Guide

This Laravel project is a sample game operator integration for Prime Mac Games.
It shows the two directions of communication:

1. The operator calls the provider read APIs to discover games and launch URLs.
2. The provider calls the operator wallet APIs while a player is in a game.

## Operator Config

The operator public ID must come from the game provider. It should not be
generated by the operator system.

Current sample value:

```text
PRIME_MAC_OPERATOR_PUBLIC_ID=3a6de854-339c-4668-ab69-cad5e168a231
PRIME_MAC_PROVIDER_CODE="Prime Mac Games"
PRIME_MAC_PROVIDER_CODE_AUTO_SYNC=true
PRIME_MAC_PROVIDER_CODE_SYNCED=false
PRIME_MAC_PROVIDER_BASE_URL=https://api.primemacgames.com
```

The operator should not invent `PRIME_MAC_PROVIDER_CODE`. On startup, this
sample can read it from:

```text
GET https://api.primemacgames.com/api/portal/operators/{operatorPublicId}
```

After a successful response, the app saves `providerCode` to `.env` and
`operator_provider_config`.

Manual sync command:

```bash
php artisan prime-mac:sync-provider-code --force
```

## Launch URL Flow

When a player clicks Launch POKER:

1. The operator creates a short-lived launch token in its own database.
2. The operator calls the provider launch-config API:

```text
GET https://api.primemacgames.com/api/portal/operators/{operatorPublicId}/games/{gameId}/launch-config
```

3. The provider returns `launchUrlFormat`.
4. The operator replaces `{launchToken}` with the actual launch token.
5. The operator redirects the player to the final game URL.

Important files:

```text
app/Http/Controllers/OperatorController.php
app/Services/ProviderApiClient.php
```

Current POKER launch format:

```text
https://livekit.poker.goscanqr.com/?launchToken={launchToken}&operatorPublicId=3a6de854-339c-4668-ab69-cad5e168a231
```

## Required Signed Headers

Every provider wallet request must include:

```text
X-Signature
X-Timestamp
X-Nonce
X-Provider-Code
```

`X-Provider-Code` must match:

```text
Prime Mac Games
```

Signing message:

```text
{timestamp}.{nonce}.{HTTP_METHOD}.{pathAndQuery}.{rawBody}
```

Signature:

```text
Base64(HMACSHA256(signingSecret, signingMessage))
```

The sample validates this in:

```text
app/Services/ProviderSignatureValidator.php
```

## Endpoint Responsibilities

### POST /api/player/authorize

Input:

```json
{
  "launchToken": "GUID"
}
```

Purpose:

The provider uses this endpoint to exchange a launch token for player details.
The operator returns player ID, display name, currency, balance, game ID, and
operator public ID.

### POST /api/balance/get

Input:

```json
{
  "playerId": "GUID"
}
```

Purpose:

Returns the latest player wallet balance.

### POST /api/bet/place

Input:

```json
{
  "playerId": "GUID",
  "gameId": 1,
  "roundId": "round-001",
  "transactionId": "bet-001",
  "amount": 100,
  "currencyCode": "PHP"
}
```

Purpose:

Debits the player's wallet. The amount must be positive.

Rules:

- Reject if balance is insufficient.
- `transactionId` must be idempotent.
- Same `transactionId` and same payload should return the original result.
- Same `transactionId` and different payload should be rejected.

### POST /api/bet/settle

Input:

```json
{
  "playerId": "GUID",
  "gameId": 1,
  "roundId": "round-001",
  "transactionId": "settle-001",
  "amount": 150,
  "currencyCode": "PHP",
  "playerActivity": {
    "result": "win"
  }
}
```

Purpose:

Credits a positive payout amount to the player's wallet.

### POST /api/bet/rollback

Input:

```json
{
  "playerId": "GUID",
  "gameId": 1,
  "roundId": "round-001",
  "transactionId": "rollback-001",
  "originalTransactionId": "bet-001",
  "amount": 100
}
```

Purpose:

Reverses a previous debit or credit.

Rules:

- If the original transaction was a debit, credit the amount back.
- If the original transaction was a credit, debit the amount back.
- Rollback transaction IDs must also be idempotent.

## Database Design

This sample uses PostgreSQL stored procedures for wallet operations. This keeps
balance updates atomic and easier to audit.

Important tables:

```text
operator_players
player_wallets
player_launch_tokens
wallet_transactions
provider_request_nonces
operator_provider_config
```

Important stored procedures:

```text
sp_provider_code_sync
sp_launch_token_create
sp_launch_token_authorize
sp_balance_get
sp_provider_nonce_record
sp_wallet_bet_place
sp_wallet_bet_settle
sp_wallet_bet_rollback
```

## What Another Operator Should Replace

An operator integrating their own system can keep the API contract and replace:

```text
app/Services/OperatorStore.php
```

with their own user, wallet, and transaction database logic.

They must keep:

- The five required endpoint paths.
- Signed request validation.
- Positive amount handling.
- Atomic wallet balance updates.
- Idempotency by `transactionId`.
- Rollback by `originalTransactionId`.
